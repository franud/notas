El que farem es el següent:
1. Instalarem la versió 14 de java des de el nostre package manager

2. Mirarem quina versió de java es té per defecte actualment al nostre sistema, per si s'ha canviat al instalar la versió 14, aixó ho podem veure de la següent manera:
	2.1 Fem ls -la /usr/bin/ | grep "java"
	2.2 La sortida mostrará que tot l'englomerat de executables de Java (java, javac, javadoc, etc.) fan un softlink a executables en un altra carpeta, que en aquest cas sería /usr/lib/jvm/default que a la seva vegada es un softlink a la versió especifica de java (11 o 14)
	2.3 Mitjançant eines que proveeixen molts sistemes operatius, com Debian (update-alternatives --config java (també hi ha una versió més handy que es update-java-alternatives) o Arch Linux (amb archlinux-java), tenim un programa en el que nosaltres podem escollir quina versió de java es té per defecte el softlink de java al /usr/bin, de forma que podem cambiar-lo de forma més fácil i per tant menys propensa a errors manuals.
En cas de voler utilitzar en particular les versions propietaries de la màquina virtual de java com la versió d'Oracle, es podria utilitzar la mateixa idea dels softlinks pero instalant aquestes versions propietaries en altres directoris per evitar colisions.
	2.3 Posarem per defecte la versió 11 de Java que ja estavem utilitzant per l'aplicació de facturació per evitar errors (a Debian no es canvia aixi que no fa falta).

3. Ara, podem utilitzar una d'aquestes dues opcions:
	Opció 1: Crear un soflink de java14 al nostre /usr/bin, i que per executar l'aplicació es tingui que fer java14 [executable java]. Aquesta es la millor opció.
	Opció 2: Crear un script per substituir executable del programa que: modifiqui la variable d'entorn PATH posant al principi el path de la versió 14 de java i després executi java [nom executable], pero per utilitzar aquesta en la mateixa sesió de shell tindria que obrir primer el de java11 i després la de java14
	
Si l'usuari en particular no es técnic i utilitza l'entorn grafic, podem utilitzar els arxius .desktop usats per l'entorn grafic per posar allí el PATH corresponent a cada aplicació de forma transparent a l'usuari.

4. He investigat i hi ha alguns programes que utilizen la variable d'entorn JAVA_HOME, variable que té el path de la nostra carpeta on son les eines de java, per si de cas es podría afegir al nostre script de la opció 2 (no fa falta afegirla a /etc/profile o a un script dins de /etc/profile.d/ ja que la nostra aplicació de facturació ja funcionaba sense).

5. 
- La forma de migrar l'aplicació vella per la nova es primer probar que no es trenqui cap part important de la lógica de negoci en altra maquina, ja que poden haber altres aplicacions i scripts que utilitzin la aplicació vella i la nova potser no es backwards compatible. 
- Després hauriem de buscar usuaris i fer-los probar aquesta aplicació nova durant uns dies i que ens donin feedback i actuar de forma adient al seu feedback. 
- En cas de decidir cambiar una aplicació per altra, primer s'hauria de avisar que es cambiará de versió amb una llista de canvis a tots els usuaris, i donar opció a probarla, on es rebrá més feedback i s'haurá de actuar de forma adient al cas, al cap del temps es cambiará totalment la nova per la vella per defecte, sense esborrar la vella.
- Es posará un sistema de monitorització que ens dirá qui utilitza la versió vella, si arriba un moment que ningú l'utlitza, es pot considerar borrarla.
